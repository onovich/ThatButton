# Phase 0 Playtest Record

Date started: 2026-06-23
Phase: Phase 0 - Baseline And Playtest Criteria

This document is the source of truth for baseline playtest evidence before changing the difficulty curve. Do not use it to redesign the full curve; use it to record what the first session actually feels like.

## Acceptance Targets

- First 5 levels are learnable: a new player can explain the rule format and recover from at least one missed read.
- First 10 levels feel winnable: a player can reasonably expect progress through level 10 after one or two retries.
- Failure invites retry: after a wrong click or timeout, the player understands why they failed and is willing to restart.
- Debug reproduction exists: confusing rounds can be reproduced with a fixed seed or equivalent recorded generation data.
- Ordinary play remains direct: without debug parameters, the game starts with normal random generation.

## Debug Seed Entry

Add `?seed=<label>` to the local or Pages URL before starting a run. Example:

```text
http://127.0.0.1:5173/?seed=phase0-level1
```

When `seed` is absent, the game keeps using normal browser randomness. When `seed` is present, the same seed can replay the same generation sequence. The debug preview helper simulates that sequence up to the requested level, then returns the generated button grid, fatal count, and rule text for design review.

For automated checks or console inspection, use:

```js
window.__THAT_BUTTON_DEBUG__.previewSeededLevel('phase0-level1', 1)
```

For design/debug sessions, add `&debug=1` after the seed or `?debug=1` without a seed. Debug mode keeps ordinary UI unchanged and writes structured round/failure events to the browser console. The same events can be inspected with:

```js
window.__THAT_BUTTON_DEBUG__.getLog()
```

## First 10-Level Review Checklist

- Levels 1-5: player identifies that the clue describes fatal buttons, not safe buttons.
- Levels 1-5: at least one success is possible without memorizing the templates.
- Levels 6-10: AND/OR/NOT rules are recognized as rule changes, not as wording mistakes.
- Levels 6-10: timer pressure feels tense but not arbitrary on desktop.
- Failure screen: player can connect the failure to the current rule text.
- Restart: player retries without needing a page refresh.

## Phase 1 Input Notes From Current Evidence

- Current baseline grid size is `3x3`; Phase 1 should decide whether early levels need a smaller grid only after human playtest evidence.
- Current generated fatal count is bounded by rule generation: early levels target 1-3 fatal buttons, later levels may allow up to 4.
- Current timer starts at 15000 ms for level 1, then tightens by level progression; Phase 1 should compare first-failure levels against this timer pressure.
- Current failure evidence can distinguish `wrong_click` from `timeout` through the debug log, but human quit reasons still require manual session records.
- Current copy ambiguity evidence is not yet collected from real players; Phase 2 should wait for recorded `ambiguity_note` entries.

## Data For Later Phases

Use these Phase 0 records as inputs to Phase 1:

- Highest level reached before first quit.
- First failure level and failure type.
- Fatal button count per level.
- Time limit at failure or near-failure.
- Number of safe clicks required per level.
- Whether the player failed because of rule complexity or timer pressure.

Use these Phase 0 records as inputs to Phase 2:

- Exact rule text that caused hesitation or wrong inversion.
- Whether the player read the clue as "press this" instead of "avoid this".
- Repeated terms that feel inconsistent, noisy, or too long under pressure.
- Failure feedback that did or did not explain the mistake.
